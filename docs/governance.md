# GOML Governance

## Overview

GOML follows an open governance model to ensure the format evolves in a transparent and community-driven manner.

## Principles

1. **Simplicity First** - Changes must not add unnecessary complexity
2. **Backward Compatible** - New versions must read old files
3. **Community Driven** - Major changes require community input
4. **Transparent** - All discussions and decisions are public

## Versioning

GOML follows Semantic Versioning:

- **Major** (1.0, 2.0): Breaking changes (rare, requires migration)
- **Minor** (1.1, 1.2): New features, backward compatible
- **Patch** (1.0.1, 1.0.2): Bug fixes only

## RFC Process

For significant changes:

1. **Proposal** - Open issue describing the change
2. **Discussion** - 30-day public comment period
3. **Vote** - Core maintainers vote (2/3 majority required)
4. **Implementation** - Accepted changes implemented
5. **Release** - Changes released in next version
